## Context

Today, when an Etsyweb PR linked to a XORCH issue merges, the Jira issue does not auto-transition to Done. Glean confirms there is no Etsy-wide convention (no smart-commit syntax, no `Closes XORCH-N` mapping); the Jira-GitHub app only auto-links commits/PRs to issues, it doesn't change status. Per-Jira-project automation rules *can* do this, but must be set up on the XORCH board.

Reference: Etsyweb docs and #devsupport thread surfaced via Glean (2026-05-06):
- https://docs.etsycloud.com/etsyweb/code-review/
- https://etsy.slack.com/archives/CRCSSEX8X/p1726058984383239

## Goal

Investigate and (if appropriate) configure a Jira automation rule on the XORCH board that transitions an issue from "In Review" to "Done" when its linked PR merges.

## Requirements / guardrails

- **Only fire on merge into the repo's default branch (`main`)**, not feature branches or stacked-PR intermediaries. Stacked-PR workflows (e.g. XORCH-742 → XORCH-740) commonly have intermediate merges that must not close the parent ticket.
- **Only transition from "In Review"** to "Done", not from arbitrary statuses. Guards against accidentally closing tickets that were never reviewed.
- Multi-PR tickets: confirm behavior when one Jira issue has 2+ linked PRs (e.g. follow-up bugfix PR after the original lands). The rule should not prematurely close.
- Document the rule and its limits in a XORCH wiki/runbook so the team knows what to expect.

## Acceptance criteria

- [ ] Investigated Jira automation rule capabilities (Jira-GitHub trigger, branch filter, status condition)
- [ ] Decision recorded: enable the rule, or document why we're not
- [ ] If enabled: rule live on XORCH board, tested with a real PR merge, documented for the team
- [ ] If not enabled: rationale captured in the XORCH wiki / this issue's resolution comment
